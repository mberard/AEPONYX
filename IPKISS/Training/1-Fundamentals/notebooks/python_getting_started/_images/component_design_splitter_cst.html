Introduction¶

This notebook illustrates design and optimization of a custom passive component within the IPKISS framework.

The objective of this notebook is to design a 1x2-splitter with a 50:50 percent splitting ratio. In order to do this 3 steps are needed:

  1. Creation of a PCell for a splitter. The splitter should be parametrized in such a way that it can be optimized for performance by varying those parameters.
  2. Optimization of the splitter: Using intuition, a fast and approximate simulation and optimization method for obtaining a good splitter is proposed. In this notebook we will use CAMFR as a solver.
  3. The proposed splitter is verified using an accurate 3D-FIT/FDTD simulation. An accurate s-matrix model is created for the splitter. In this notebook we will use CST Microwave Studio as an engine for this FIT/FDTD simulation and the s-matrix extraction.

Step 3: Automated opmization using python's optimization libraries.¶

We can now use the standard python optimization libraries to optimize the splitter. The function min_function(x) that is defined below takes in a vector x that is mapped onto the layout properties we want to optimize. It returns the minus of the transmission in dB which will be used as an objective function we want to minimize.

We use the manually obtained values as a starting point for our optimization (Hint values are used here by default)

Step 4: 3D-FIT simulation with CST and creation of an accurate S-matrix (beta)¶

It is possible to export any IPKISS virtually fabricated structure to CST Microwave studio and import the simulated S-Matrix back into IPKISS.

S-Matrices returned by CST are accurate on a broad spectrum as their FIT solvers take into account:

  • Material dispersion.
  • Modal shape dispersion.
  • The roundness of boundaries.

<img src="_images/cst_flow.png" width=600>

Simulation flow step 1: Export your structure to CST.¶

The simulation flow allows for the export of a virtually fabricated structure to CST. To do this IPKISS creates a .mod file that can be read by CST.

The following information of your component is correctly transferred.

  1. Complete geometry: The virtually fabricated layout as defined by the virtual fabrication rules in your TECHNOLOGY.
  2. Materials: The optical properties of your materials as defined in your TECHNOLOGY.
  3. Ports: The location and orientation of your ports.

A simulation with meaningful defaults is setup within CST. The following settings are set automatically but can of course be overruled manually.

  1. Grid: The automatic meshing algorithms are used. You can set the bounding box of the mesh from IPKISS.
  2. Port excitation: The ports are set to be used as sources with settings that are best suited for S-Parameters extraction.
  3. Field Monitors: A field monitor is added for the entire structure but other field monitors can be added.

Fully scripted simulations¶

What you have seen above is a one way-shot, meaning that you manually launch modify and lauch the simulation. For optimization purposes it may be best to do everything scripted. The good news is that this is possible too!

To achieve that goal one must do the following:

  • Keep track of the modifications you made in the GUI.
  • Start the simualtion and extract results from script from script.

We now will go over those steps.

Keep track of the modifications made in the GUI.¶

As you interact with the GUI of CST, CST will add VBA commands to the history list. To avoid having to do these manipulations each time you can fetch those commands from your CST project as shown in the video below.

<video controls src="_videos/cst_save_additional_commmands.mp4" width=900>

The file (additional_commands.mod) was obtained in the GUI by adding port symmetries, changing mesh_settings and reduce the solver accuracy.

Start the simulation and extract results from script from script.¶

In order to run a CST simulation in batch mode, we will first create a model including the additional commands saved in the file additional_commands.mod as well as the postprocessing steps which we want to be performed after each simulation run. The postprocessing steps are defined via VBA code in the file postprocessing_commands.mod. Here we just save the S-Parameter in a touchstone file.
